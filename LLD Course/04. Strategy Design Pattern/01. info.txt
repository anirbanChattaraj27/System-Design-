QS.) WHY Inheritance is so BAD ....?

Diagram: refer image_part1
    Suppose there is parent class Robot which has methods walk, talk, projection. (projection will change robot to robot but walk talk will be common in every robot)
    - and child classes inherited Robot class and companionR, WorkerR these 2 classes will @Override projection method from Robot class. so projection method will keep on chaning, dependes upn new child class
    - NOW, let say there is another robot sparrowR whcich also inherits Robot class and this class will have 1 different methods like fly() method, so along with walk, talk and projection there is another method fly.

Diagram: refer image_part2
    NOW, there are 2 new robot comes which also have this fly method so I have to repeate fly() method for multiple types, and this breaks DRY(Do Not Repeate Yourself) principal. 
        - And if I put this fly method inside Robot class then other classes will also extends fly method but I want this fly method only to specific robots not all, so I will change the code structure
        - AND, I will create another child class called flyableR and now I can define the fly() method and in SparrowR robot i will just call projection method
        - NOW, I can extend n number new robots which can fly and inherits that from flyableR

        - BUT.... what if 10 robot comes again so in that scenario I will have to call that fly() nethod 10 times and reapeate the same code so again  it will not follow DRY principal

***************SOLUTION TO Inheritance is NOT more Inheritance**************


*******************************************************************************************************************

PROBLEMs with Inheritance:
    1) did not get code re-use much
    2) to add new feature, lot of changes were made
    3) break Open-Close-Princiap (OCP)


SOLUTION to above probelm: 
Strategy Design Pattern: it Defines a family of Algorithm, put them into separate classes, so that they can be change at runtime

*******************************************************************************************************************


Diagram: refer image_part5
    what design patterns say is some part of code will change continuesly and some part of code remains constant, so we have to always have to separate our DYNAMIC code with the STATIC code which doesnt get change
    here in Robot class talk, walk, fly is changing continuesly but projection remains same every time

    - to solve this problem, I can remove walk, talk, fly from the robot class, and create separate interfaces, and I will use them using robot class
    I will create fly, talk, walk methods using the interfaces 
    and I will store the references of that interfaces in thr Robot class

        Now I can create numorus number of robots with different features and to create it I do not have to change  Robot class

        - In this image_part5, here 1 is client and other classes are called strategy(defination of strategy pattern says, it Defines a family of Algorithm, put them into separate classes, so that they can be change at runtime)
